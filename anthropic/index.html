§ 02

Three problems I'd work on.

01
Hallucination & Reliability

LLM hallucination is still caught by users, not systems.

Most evaluation pipelines check correctness at launch. In production, the model drifts — domain distribution shifts, prompts evolve, edge cases accumulate. The current state asks users to be the hallucination detector, which is exactly who shouldn't be. I've built the continuous alternative: automated evaluation pipelines that track faithfulness, relevancy, and grounding on every deploy, not just before it.

Approach
  • Instrument live Claude interactions with lightweight faithfulness scoring, sampled continuously.
  • Build a drift detection layer that surfaces when a model's failure profile changes, not just when it's wrong on known evals.
  • Design a feedback loop from flagged outputs back into the eval set — evals grow with the product, not behind it.
Why I can say this

Built production hallucination tracking for ComplyStream's FinCrime platform — real-time detection across 200+ analysts' workflows. Integrated into CI/CD as a deployment gate. Reduced false-positive compliance flags by 40%.

02
Agentic Systems

Agents scale their mistakes as readily as their output.

When an agent acts on bad intermediate output, subsequent steps build on that error — silently, confidently, at speed. Claude Code and Computer Use are powerful, but the failure mode I keep noticing is the same one I've had to engineer around in autonomous agent systems: the absence of a self-correction layer. An agent should know when it's off-track before the user does.

Approach
  • Add lightweight post-action verification: every consequential tool call has a condition it should satisfy — the agent checks it, not the user.
  • Build structured recovery paths for the most common failure types (tool returns unexpected schema, action has no observable effect, output contradicts prior state).
  • Instrument agent traces so failure points are visible, not buried in logs — a prerequisite for improving them.
Why I can say this

Architected multi-agent FinCrime systems at ComplyStream — agentic orchestration over financial records with 3x workflow efficiency improvement. Self-verification was the hardest part; it's also the part that made everything else trustable.

03
Infrastructure Cost & Scale

Token cost shapes who can afford to build on Claude.

Inference cost is a product decision, not just an engineering one. At ComplyStream, I reduced token overhead by 30% without degrading model output quality — using structured serialization, prompt compression, and caching strategies that preserved semantic content. The teams building on top of Claude's API face this problem at every design decision. Making it easier to be efficient without sacrificing capability is work worth doing at the platform level.

Approach
  • Document and systematize the token-efficiency techniques that work in production — not theoretical, tested against real workloads.
  • Build tooling that helps developers profile their prompt token consumption and identify reduction opportunities.
  • Design a reference architecture for high-frequency LLM API usage that integrates cost controls without requiring every team to rediscover the same optimizations.
Why I can say this

Achieved 30% token reduction on ComplyStream's AWS Bedrock workload using BSON serialization, prompt engineering, and response caching. Also cut API latency by 50% and infrastructure costs by 50% at Diverge — same problem, different stack.
